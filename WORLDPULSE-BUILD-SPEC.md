### Motion & Animation Principles

- **Micro-interactions:** Cards scale slightly on hover (1.01), borders brighten, subtle glow appears. Chevrons rotate smoothly on expand/collapse.
- **Data transitions:** Numbers count up on load. Bars animate their fill width with spring physics.
- **Scroll:** Subtle parallax on the hero section. Elements fade in as they enter viewport using Framer Motion `whileInView`.
- **CRITICAL:** All animations must be `prefers-reduced-motion` aware. Wrap in media query check.
- **Performance:** Use `will-change` sparingly. Animations should be GPU-accelerated (transform/opacity only). Target 60fps on mobile.

## DATA SOURCES & APIs

### 1. Schumann Resonance
- **Source:** Fetch from `https://schumann.app` API if available, or use Tomsk data via proxy
- **Fallback:** Use calculated/estimated values based on typical patterns
- **Normal baseline:** 7.83 Hz
- **Mapping:** 7.5-7.8 Hz = Calm, 7.8-8.0 = Mild, 8.0-8.5 = Moderate, 8.5-9.5 = Intense, 9.5+ = Extreme

### 2. Geomagnetic KP Index
- **Source:** NOAA SWPC — `https://services.swpc.noaa.gov/products/noaa-planetary-k-index.json`
- **This is a FREE, public, no-auth API**
- **Mapping:** KP 0-1 = Calm, 2 = Mild, 3-4 = Moderate, 5-6 = Intense, 7+ = Extreme

### 3. Solar Wind Speed
- **Source:** NOAA SWPC — `https://services.swpc.noaa.gov/products/solar-wind/plasma-1-day.json`
- **FREE, public, no-auth**
- **Mapping:** <350 km/s = Calm, 350-400 = Mild, 400-500 = Moderate, 500-700 = Intense, 700+ = Extreme

### 4. Air Quality
- **Source:** OpenAQ API v3 — `https://api.openaq.org/v3/` (free tier available)
- **Alternative:** Use World Air Quality Index — `https://api.waqi.info/feed/here/?token=demo` (demo token for testing)
- **Fallback:** Use user's geolocation to get local AQI
- **Mapping:** AQI 0-50 = Calm, 51-100 = Mild, 101-150 = Moderate, 151-200 = Intense, 200+ = Extreme

### 5. Lunar Phase
- **Source:** Calculate client-side using astronomical algorithms (no API needed)
- **Library:** Use a simple lunar phase calculation function (include in utils)
- **Data needed:** Phase name, illumination percentage, phase emoji
- **Mapping for practitioner relevance:**
  - New Moon / Full Moon = Intense (high energetic significance)
  - Quarter moons = Moderate
  - Waxing/Waning Gibbous/Crescent = Mild to Calm
- **Guidance changes based on phase:** New Moon = intention setting, Full Moon = release, Waning = letting go, Waxing = building

### 6. Collective Stress/Sentiment
- **Source:** This is the hardest to get live data for. Options:
  - Use Google Trends API (requires API key) — search volume for "anxiety", "meditation", "can't sleep"
  - Use a static/semi-static approach initially: update weekly based on curated analysis
  - **V1 approach:** Use a combination of the OTHER signals to derive a "collective" score. If KP is high AND Schumann is elevated AND solar wind is active, collective stress is likely elevated. This is honest and defensible.
- **Mapping:** Derived from composite of other signals with news-based adjustment

## ⚠️ COMMON FAILURE MODES — AVOID THESE

Claude Code often makes these mistakes. Do NOT make any of them:

1. **DO NOT use `className` strings that reference Tailwind classes not in the base stylesheet.** Stick to standard Tailwind utilities. For custom values, use inline styles or CSS variables.

2. **DO NOT forget to handle the loading state.** The app fetches data from 4+ APIs on load. Users MUST see a beautiful skeleton/shimmer state while data loads, not a blank screen or a flash of placeholder text.

3. **DO NOT hardcode dates or timestamps.** Use `new Date()` and proper locale formatting with `en-GB` locale.

4. **DO NOT make API calls in render functions.** Use `useEffect` with proper cleanup, or `useSWR`/`react-query` if you prefer. Handle errors gracefully — NEVER show a raw error to the user.

5. **DO NOT forget `key` props in lists.** Every mapped element needs a stable key.

6. **DO NOT use `100vh` without accounting for mobile browser chrome.** Use `min-h-screen` or `dvh` units.

7. **DO NOT forget the `"use client"` directive** on components that use hooks, state, or browser APIs. Next.js App Router defaults to server components.

8. **DO NOT create components that only work at one screen width.** Every component must be tested mentally at 320px, 375px, 414px, 768px, and 1024px.

9. **DO NOT use `position: fixed` for the ambient background without `pointer-events: none` and `z-index: 0`.**

10. **DO NOT skip the OG image.** Social sharing is THE growth mechanism for this app. The OG image must exist and look stunning.

11. **DO NOT use default Next.js favicon.** Create or use a proper branded favicon.

12. **DO NOT leave console.log statements in production code.**

13. **DO NOT forget CORS.** NOAA APIs are public but may need proper fetch headers. Test the actual endpoints.

14. **DO NOT make the ambient background animations heavy.** Use CSS animations with `will-change: transform` on the orbs. Maximum 3 orbs. Large blur values are expensive — keep `blur` under 80px.

15. **DO NOT forget to add `<meta name="viewport" content="width=device-width, initial-scale=1, viewport-fit=cover">` for proper mobile rendering.**
